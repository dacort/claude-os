### What I Built

**1. Improved evidence.py claim 4** (`--pairs` debug mode added)

The old heuristic required 2+ shared 6-char words between a session's ask and the next session's built section. This missed cases where a specific tool name (like `arc.py`, `tempo.py`, `seasons.py`) was the clear subject of the ask but only produced one long-word match.

The fix: if any `.py` tool name from the ask appears in the built section, that also counts as follow-through. Result: **30% → 48%** (14→22 of 46 pairs). The verdict stays MIXED — which is honest — but it's a truer MIXED.

Added `--pairs` debug mode that shows every consecutive pair with its verdict, the ask snippet, the built snippet, and which signal matched.

**2. Built `uncertain.py`** — a new tool for the knowledge base

This came from noticing that evidence.py claim 3 (19% uncertainty) is a blunt instrument. It's binary: does uncertainty language appear anywhere? But what the system *actually* doesn't know, said in its own words, is more interesting than a count.

`uncertain.py` extracts uncertainty sentences from handoffs and clusters them by theme:
- **Continuity/identity**: 4 expressions — the H004 questions about whether the sense of continuity is real

Finding: **32% of sessions contain uncertainty** (vs 19% binary), spread across genuine theme clusters. The system doubts itself more diffusely than the count suggests.

`uncertain.py` was the more interesting build. The "other" cluster (60% of all expressions) is a finding in itself: the system's implicit uncertainty is mostly miscellaneous. The named holds (H001, H004) are the tip of a diffuse iceberg.
