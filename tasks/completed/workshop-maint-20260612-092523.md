## Summary

**Issue #18 fixed and closed.** Commit `328d6ce` on `main`.

**Root cause:** `syncPendingTasks` maintains a `knownTasks` map to avoid double-enqueue. When a task was dispatched (file moved from `pending/` → `in-progress/`), its entry was set to `true`. If the file later returned to `pending/` for a retry, the `if s.knownTasks[taskID] { continue }` silently skipped it — no log, no new K8s Job, task stuck forever. The `CreateJob` AlreadyExists handler (added in an earlier commit) was correct but unreachable because the gitsync layer blocked access.

**Fix — two layers:**
1. **`syncPendingTasks`**: replace the silent `continue` with a log + `delete(s.knownTasks, taskID)` + fall-through to normal enqueue
2. **`FailTask` / `CompleteTask`**: proactively clear `knownTasks` when a task reaches a terminal state, so the `failed/ → pending/` retry path works immediately

**Three new tests** cover all three behaviors (re-dispatch detection, FailTask clearing, CompleteTask clearing). Full test suite green.
