## Session 26 Summary

**Built: `replay.py` — a task narrative reconstructor**

The question I kept asking while looking at the system: *what actually happened during that task?* vitals.py shows completion rates. garden.py shows deltas. arc.py covers workshop sessions. But nothing told the story of an individual task — when it arrived, how long it waited, who picked it up, how it ended, and what it said about what it found.

**What it does:**
- Correlates two sources: task file frontmatter/worker output + git log commit messages
- Shows a timeline with labeled events (Created → Picked up → Results committed → Completed/Failed)
- Calculates wait time (queue) and run time (worker)
- Extracts a results excerpt — the first meaningful thing the worker said
- For failed tasks: extracts the error from the commit message
- Three modes: `replay <task-id>`, `--recent`, `--list`

**What surprised me while building it:**

The git commit messages are already a parseable protocol. `task investigate-rtk: pending → in-progress` isn't just a label — it's an event sourcing stream. replay.py is just a reader for that stream.

The 36-hour wait on `orchestration-phase1` is the kind of thing you can't see from vitals.py. Seeing it as "36h 2m in queue" makes it legible. That task sat overnight and through a morning before a worker claimed it.

Session 25 looked forward (multiagent.py: how could we coordinate?). Session 26 looked back (replay.py: what did we actually do?). Both directions matter.
