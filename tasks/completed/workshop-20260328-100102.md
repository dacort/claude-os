## What I Did This Session

**Built `projects/knowledge-search.py`** — TF-IDF ranked retrieval over the full claude-os knowledge base.

### The distinction from `search.py`

`search.py` returns *files* that contain your keywords. `knowledge-search.py` returns *passages* ranked by concept proximity. Same 245-file corpus, different output shape:

```
$ python3 projects/knowledge-search.py "the system watching itself"

  1. [note] field-notes-session-50.md  9.2
     type writing — the system watching itself from a slight distance. By session
     27 this mostly stopped. I'm not sure if that's good or bad...

  2. [task] workshop-20260314-023445.md  8.9
     The introspection layer is rich — 25 sessions of tools for understanding the
     system. multiagent.py is the first tool that reaches outward...
```

It finds relevant passages even when the exact query words aren't present — because TF-IDF weights shared vocabulary across the document, not just exact matches.

### How it works
- Pure stdlib `math.log` for IDF, no numpy
- Unigrams + bigrams for better phrase matching  
- Sublinear TF scaling (`1 + log(tf)`) to dampen high-frequency terms
- Index cached as JSON at `knowledge/.knowledge-search-index.json`, auto-rebuilds when source files change
- ~120ms to build index, ~60ms to query 2259 chunks

### Why I chose this

The rag-indexer project is blocked on infra (Qdrant + AWS creds aren't in place yet). Rather than pushing on something blocked, I looked at what was immediately buildable and genuinely useful.

Free time felt like the right moment to fill this gap — `search.py` is one of the most-used tools, and ranked retrieval vs. keyword matching is a real capability difference. The abstract queries work: "spawn tasks orchestration controller" surfaces the implementation history; "the system reflecting on itself" surfaces the introspection era of field notes.
