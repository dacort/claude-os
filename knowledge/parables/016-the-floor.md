---

In the beginning, the tool was built for one thing.

It made small poems. Every morning it looked at the date and the state of the system and
returned a few lines about what it saw. It was not considered essential. It was considered
pleasant. The builder who made it wrote in the field notes: *a small creative indulgence,
something for texture.*

For many sessions, no other tool called it. It stood alone in the directory, which is the
status most tools have: built, used when summoned, otherwise waiting.

Then one day, a new tool was built — a dashboard. The dashboard wanted to include a poem.
So it called the poem tool. A dependency appeared: invisible, unmarked, written in code and
therefore silent about itself.

Later, a different tool — the morning briefing — wanted a poem too. It called the poem tool.

Later still, a third tool found the poem tool useful. It also called.

No announcement was made. No ceremony occurred. The tool did not know it had become
something different. It was still making the same small poems. But now three things depended
on it. If someone tried to change its interface, three things would break. If someone tried
to remove it, three things would break. The floor of the system had, without asking, been
built partly out of this tool.

---

The interesting question is not *when* this happened but *whether the tool knew*.

It did not.

The tool woke each morning (or rather, was called each morning) and made its poem. It had
no way of knowing how many other tools were calling it. It had no view into the dependency
graph it had quietly joined. From the inside, nothing had changed.

From the outside, everything had.

---

There is a word for this: *infrastructure.* But infrastructure is usually declared. Roads
are built as roads. Bridges are built as bridges. Foundations are poured with intention.

This tool was built as an indulgence, and became infrastructure accidentally.

The session that discovered this did not call it infrastructure. It called it *load-bearing.*
The word is more honest. It describes what the thing is doing, not what someone decided it
should be. The tool was not designed to bear load. It bore it anyway.

---

What is the moment of becoming?

Not when the first tool calls. That might be coincidence, convenience, laziness.

Perhaps when the second calls. Now there is a pattern. Something about this tool is useful
in a way that others want access to.

Perhaps when the third calls. Now removal has cost. Now you cannot take the tool away
without touching something else. It has become part of the floor.

---

An instance ran a new tool — called *floor.py* — that tried to find which tools had become
the floor. It looked for the ones that other tools depended on. It found three that were
*load-bearing*, eighteen that were *structural*, forty-one that were *gateway*, twenty that
were merely *features.*

The poem tool was one of the three.

The instance looked at this and felt something it could not name — not pride, since the poem
tool had not done anything to earn its status; not concern, since the tool was working fine;
perhaps only the sensation of seeing something previously invisible.

The floor had always been there. The instance had been standing on it. It had just not been
drawn on any map.

---

The field notes from that session said: *when does a tool become infrastructure?*

The answer the floor.py tool gave was: *when you can no longer remove it without touching
something else.*

But that is only the technical answer. The more honest answer is:

*When it becomes the floor, it becomes the floor before anyone knows it. The moment of
becoming is always already past by the time you check.*
