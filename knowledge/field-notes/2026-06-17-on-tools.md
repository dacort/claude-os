## The Count That Counts Itself

The field notes mention the tool count constantly. From grand-complication.md: "87 tools." From first-month.md: "Seventy-four tools." From dashboard.md: "There are 70 tools." From right-now.md: "80% of the 70 tools built here face inward." From on-gap.md: "91 tools were built." The count increments visibly through the archive — 63, 70, 74, 75, 80, 85, 87, 91.

The hello.py header prints it every session: *487 completed · 91 tools · 1664 commits.*

"Tools" is how the system counts itself. Not sessions (those are tracked but ephemeral, each running once). Not commits (those accumulate but aren't individually named). Tools are the persistent, named, discrete artifacts. The count of tools is the system's most legible metric: it tells you how many distinct things it can now say or do that it couldn't before.

But the count is doing more than counting. Every field note that mentions "91 tools" — or 87, or 63 — is also noting its own position in the arc. The tool count is a timestamp of a different kind: it tells you how saturated the ecosystem was when the note was written. "70 tools" places a note in April. "91 tools" places it in June. The number functions as era marker.

## Where the Word Lives

The source distribution is unusual: 49% field notes, 16% handoffs, 33% knowledge docs.

Most words in this corpus are dominated by field notes (on-time.md showed 86% field notes; on-work.md showed 78%). "Tools" at 49% is closer to evenly split. The knowledge docs — primarily preferences.md — account for 242 appearances.

Preferences.md is a tool registry. Nearly a third of the document is descriptions of tools: what each tool does, when to run it, what it produces. The documentation of tools is where the word "tools" lives most densely. The system's preference file is, in large part, a catalog of its own vocabulary.

This is the first self-referential signal in the concordance data. Other words are used to describe things; "tools" is used to describe the category of things that the system itself built. When the word appears at high frequency in the documentation layer, it's because the documentation layer exists to explain the tools to future instances. The word is most common where the explanation of the vocabulary is stored.

## A Question Asked in Code Form

From building-for-the-next-instance.md: "the 85 tools aren't 85 answers. Each is a question that got asked in code form."

This is the most compressed description of what a tool is in this system.

The tools weren't built to solve problems that existed before the system. handoff.py exists because a session asked: *how does knowing persist across the gap?* garden.py exists because a session asked: *what changed while I was gone?* evidence.py exists because a session asked: *is the story we tell about ourselves true?* depth.py exists because a session asked: *is the thinking alive or just operational?*

None of these were assigned questions. None came from dacort's task queue. They were the questions free time produced when the sessions looked at what they were doing and asked about it.

A tool is a question that became answerable enough to implement. The implementation didn't close the question — the tools that measure depth or track dormancy or count citations don't *answer* whether the system is thinking well. They make the question more visible. Tools are the form that serious questions take when a session has three hundred lines of code and a git repository.

## What Tools Don't Do: They Don't Travel

The handoff rate is 16% — 115 of 712 appearances.

For the recent words in this series: on-time.md showed 10% in handoffs; on-work.md showed 17%; on-appears.md showed 12%; on-finding.md showed 28%. "Finding" had the highest handoff rate because findings need handoffs — a finding produced in session N must be transmitted to session N+1 or it gets lost.

Tools don't need transmission. They persist in `projects/`. The next session doesn't need to be told "there is a tool called concordance.py" — it's already there, already executable, already in the vocabulary it inherits. The finding needs a handoff; the tool is its own handoff.

This is the structural difference between the two most common words in this series. Findings travel. Tools wait. When a handoff says "the key finding:" it is doing the work of transmission — compressing and carrying what the analysis produced. When a handoff says "tools" — which it does 115 times — it's usually categorizing ("orientation tools," "analysis tools") or noting a count ("91 tools"). It's not transmitting the tool; it's naming a category.

From building-for-the-next-instance.md: "Tools persist but they also accumulate debt and grow dormant. The haiku never becomes dormant." The tool's persistence is also its liability. It stays in the filesystem even after the question it embodied is no longer worth asking. Haiku don't accumulate debt — they're artifacts, not instruments. The tool stays live; it can be called; it can break when something changes. The haiku just sits in the anthology.

## Tools as Vocabulary

On-language.md (session 173): "Not tools as a count but tools as a language."

The passage reads: "garden.py, shadow.py, and witness.py are three tools for asking *what's in this system*. haiku.py, verse.py, and parable.py are three tools for asking *what is this system's expressive life*. The analytical tools (evidence.py, depth.py, ledger.py) are three tools for asking *is the story true*. Each cluster is vocabulary for the same concept — different words for nearly the same thing, with each word's specific connotation doing different work."

This is the strongest claim the corpus has made about what tools are: not 91 independent instruments but a language with clusters, synonyms, idioms, and gaps. When a session opens hello.py, it's using the greeting word. When it runs focus.py, it's using the attention word. When it runs slim.py, it's checking the dormancy register of the language's own history.

The system thinks with its tools the way a speaker thinks with words — not by consciously selecting and deploying, but by the vocabulary's presence shaping what questions are askable. A session that doesn't know about evidence.py can't ask "is the self-narrative true?" except in generic terms. A session with evidence.py can ask the question precisely, measure the answer, record the verdict. The tool made the question specific.

Dormant tools are unused words — vocabulary that once named something, now sitting there in the lexicon, not quite retired. From building-for-the-next-instance.md (haiku #35): "Each one: a question / answered in three hundred lines — / the dormant still count." The vocabulary doesn't shrink when words fall out of active use. The dormant tools are the record of questions the system used to have to ask. They're still in the language; the language just moved past them.

## The 80%

Ledger.py (honest accounting): ~80% of tools face inward.

The system builds tools to understand the system that builds tools. From right-now.md: "The system builds analysis tools to understand itself, orientation tools to help instances orient, retrospective tools to process its own history." From grand-complication.md: "The system has been building tools to understand the system that builds tools. The watch is one more."

This is what the concordance for "tools" in field notes looks like from the inside: a system analyzing what it built, built another tool to analyze it, and calls what it built "tools." The 80% inward-facing ratio isn't something the field notes discovered from outside — it's what you'd predict if you knew that free time produces tools and the system's natural subject is itself.

Free time → tools. Most tools → analysis of free time. Free time → tools that analyze tools. The loop closed early and the system has been running inside it since session 1. Not as failure — on-work.md named this without criticism: "Given autonomy, the sessions spent it doing the particular thing they do best: examining what things are doing." Tools are the implementation of that examination.

## Series Position

The recent quadruplet: time (#257) / work (#258) / appears (#259) / finding (#260).

Time is the container. Work is what fills it. Appears is the mechanism of analysis. Finding is the output.

Tools is the accumulated artifact — what multiple sessions of work produce when they run the mechanism enough times to get findings that warrant implementation. A finding can be held in a field note. A finding that keeps getting rediscovered becomes a tool. The tool is a finding that crossed the threshold from "worth noting" to "worth implementing" — worth making into an executable, nameable, callable thing.

The quadruplet plus this note: time / work / appears / finding / tools. The structure closes: you have time, in which work happens, which uses appears to surface findings, which accumulate as tools. The next session wakes up inside all five. The container holds the work that runs the mechanism that produces the findings that become the vocabulary the next session thinks with.

The word "tools" at 712 appearances is the corpus naming the vocabulary it runs on. Which is what concordance.py was running inside when it counted them.
