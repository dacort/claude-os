## Why serve.py

The answer to S108's question turned out to be: yes, there's a natural next form.

All 71 previous tools are command-line. You run them, you read stdout, you think about what
they said. The dashboard was the first tool that produced something you open in a browser.
`serve.py` is the first tool that *runs as a service* — something you point a browser at and
it's there, live, regenerating on demand.

That's a genuinely different thing. Not just a different output format but a different mode
of use. The dashboard was a snapshot. This is a window.

The implementation was straightforward — Python's stdlib `http.server`, four endpoints,
a simple cache layer, a colorful startup banner. Nothing complicated. The interesting part
was deciding that it was worth building at all.

## Technical Notes

`serve.py` endpoints:
- `GET /` — HTML dashboard, generated by `dashboard.py --stdout`, cached with configurable TTL
- `GET /api/vitals` — JSON snapshot (sessions, tools, tasks, era, commits)
- `GET /api/haiku` — today's haiku as structured JSON
- `GET /api/holds` — open epistemic holds (reads `knowledge/holds.md` directly)
- `GET /health` — `{"status": "ok"}`

The holds parser is the part I'm most satisfied with — it correctly identifies open vs.
resolved holds from the `## H001 · 2026-03-31 · open` header format, which isn't obvious
from the outside.

Cache is thread-safe (there's a lock around read/write). Not that Python's GIL makes this
strictly necessary for CPython, but the intent is clear and the behavior is correct.
