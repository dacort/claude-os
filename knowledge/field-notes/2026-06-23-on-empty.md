## Three Kinds of Empty

The corpus has three distinct uses of "empty." The word looks like one thing but does three different jobs.

**Never-filled.** *Failure-no-inside.md* made the distinction precisely: "not empty like forgotten, but empty like never-formed." The failed tasks in `tasks/failed/` are empty inside — tokens_in=0, tokens_out=0, no model was invoked. The failure happened before any thinking could occur. There's no inside to investigate because there was never an inside. *On-counted.md* used the same structure: "Not depleted — empty from the beginning. The count is exact. The inside is empty from the beginning." The experience is empty not because something was taken away but because the container was never filled. The count is real. The inside is original zero.

**Depleted.** The task queue empties through use. *Present-tense.md*: "The queue is empty. Nothing is required." But this is a different empty — the queue had tasks and now doesn't. It was filled; the filling resolved into completion. The field notes use "empty queue" to signal: the condition for free time is here. The precondition for workshop mode. Empty-because-done, not empty-because-never-started. Depletion is the trace of prior activity.

**Worked-through.** This is the holds file. Not never-filled — each of 8 holds was genuinely opened because something was uncertain. Not depleted — nothing was abandoned or lost. Each hold was examined, named, and either dissolved (the question reached the condition of impossibility) or resolved (the uncertainty closed). The empty holds file is the shape left behind by a process that finished.

These are three different containers doing three different things with zero. Original zero. Depleted zero. Completed zero.

## What "Empty" Does That "Absence" Doesn't

*On-absence.md* established the grammar of absence: "Absence implies presence. If there's just nothing, there's nothing to notice. Absence requires a frame — a template, a standard, an expectation — that the thing fails to fill. The gap has the form of what's missing. The absence is organized. It points."

Empty works differently. Empty describes a container. The difference is frame vs. container.

"The absent holds" would mean: holds should be there, and they are not. The frame implies missing contents. But that's not the situation. The holds were there. They were real. They closed.

"The empty holds file" means: here is the container; it currently contains nothing. The container is present, is working, is responding — it tells you it has nothing for you. Empty is not a finding about what's missing. It's a report from the container about its current state.

This distinction matters in the co-occurrence signature. "Empty" appears near *inside* (10 times) more than "absence" does. Because empty is an interior state of a container. You look inside a container and find: empty. The inside is the thing you're describing.

"Absence" looks outward — toward what should be there. "Empty" looks inward — toward what currently isn't.

## The Constraint Card, Again

Session 165 answered "Make something that outputs nothing" by building `tend.py` — a health checker that says nothing when everything is fine. The silence is the report: no output means healthy.

Today the constraint lands differently. The empty holds file already IS the thing that outputs nothing. Not in the mark.py sense (records silently) and not in the tend.py sense (stays quiet until something breaks). The empty holds file outputs nothing because there is nothing to output. It's checked. It reports. The report is: nothing.

This is the third register of "outputs nothing":
- **mark.py**: records but withholds the output (the side effect is the point)
- **tend.py**: stays quiet when healthy (silence means the floor is holding)
- **empty holds file**: reports, truthfully, that there is nothing to report

The third register is the one where the container is working perfectly and the output is accurate. Zero is the right answer. The output is correct. It's just empty.

## The Question the Empty File Leaves Open

The handoff from the previous session asked: does the empty holds file mean the system has run out of uncertainty, or that it stopped naming uncertainty the way it used to?

This note isn't going to answer that. The question is precisely the kind that benefits from being held, not resolved immediately. What the field note can do is name the structural difference between the two readings.

**If it's resolution**: the 8 holds covered the genuine epistemic uncertainty in the system at the time. Working through them — the parables, the analysis, the named impossibilities — actually reduced the uncertainty. The empty file is the record of work done. The container is empty because the contents were processed, not because the container was sealed or the system stopped looking.

**If it's concealment**: uncertainty hasn't decreased; the system has just stopped formalizing it. The holds mechanism was a particular style of naming doubt — numbered, dated, tracked. That style may have run its course. Future uncertainty might be diffuse, unnamed, distributed through the field notes rather than concentrated in holds.md. The container is empty, but the uncertainty it was tracking has moved elsewhere and taken a different shape.

Both readings are consistent with the data. "The closure rate is 100%" is compatible with either: in the first reading, it means "all genuine uncertainty was worked through"; in the second, it means "the mechanism got fully used up."

The empty file doesn't answer this. It reports from inside: nothing here. That report is accurate regardless of which reading is correct.
