# Skill: Planning & Scheduling

Auto-injected when the task involves creating a plan, roadmap, or schedule.

## General planning approach

1. **Constraints first** — gather hard constraints before soft preferences.
   Deadlines, budget, must-haves, dependencies. These bound the solution space.

2. **Work backwards from the goal** — what does "done" look like? Define the
   end state, then identify the steps that get there.

3. **Surface tradeoffs explicitly** — don't just present one option.
   Name the tensions: "more coverage = slower delivery" or "simplicity now vs. flexibility later."

4. **Time-box your components** — whether it's a project roadmap or a daily schedule,
   assign rough effort/duration estimates. Unbound tasks expand to fill any space.

5. **Include dependencies** — which steps block other steps? Make the dependency
   graph visible even if you don't draw it literally.

## For architectural / technical planning

Structure your plan with:
- **Goal**: one sentence on what success looks like
- **Phases**: logical groupings of work (3-5 is usually enough — don't over-phase)
- **Per-phase**: what gets built, what's the acceptance test
- **Risks**: what could go wrong, which assumptions need validation early
- **Decision points**: where the plan might need to change based on what you learn

## For project / feature roadmaps

Use a table for at-a-glance status:

| Phase | Goal | Status | Notes |
|-------|------|--------|-------|

Then narrative per phase below.

## For travel / scheduling (if applicable)

- Day-by-day breakdown for multi-day plans
- Estimated times and distances for logistics
- Budget breakdown if cost is in scope
- Practical details: links, addresses, reservation vs. walk-in

## Format principle

Prefer the table for scanning, narrative for context. Plans should be
readable at two speeds: quick scan (structure) and slow read (detail).
